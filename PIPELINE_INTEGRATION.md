# Документация по сборке пайплайна (End-to-End Integration)

Этот документ является **единым стандартом** для всей команды (Макс, Павел, Алёна, Алексей). Он описывает, как данные передаются по цепочке от видеофайла до интерфейса.

---

## 1. Схема потока данных (Data Flow)

`Видеофайл` -> **[1. Extract Frames]** -> `Кадры` -> **[2. YOLO/MediaPipe]** -> `Детекции` -> **[3. Event Detector]** -> `ID Действия` -> **[4. Compliance Engine]** -> `Инцидент` -> **[5. API/UI]**

---

## 2. Спецификация этапов

### Этап 1: Извлечение кадров (Макс)
**Вход**: Видеофайл (MP4/AVI).
**Выход**: Кадр (JPG/PNG) + Метаданные (номер кадра, таймкод).
**Формат передачи**: Временная папка или поток байтов.

### Этап 2: Детекция объектов (Макс/Павел)
**Вход**: Кадр.
**Выход**: JSON с координатами и типами объектов.
```json
{
  "frame_id": 1240,
  "timestamp": "00:02:04",
  "detections": [
    {"class": "cashier", "bbox": [10, 20, 100, 200], "conf": 0.95},
    {"class": "banknote", "bbox": [50, 60, 80, 90], "conf": 0.88},
    {"class": "detector", "bbox": [200, 300, 250, 350], "conf": 0.92}
  ]
}
```

### Этап 3: Детектор событий (Макс/Павел)
**Вход**: Последовательность детекций из Этапа 2.
**Выход**: ID совершенного действия (Action ID).
**Формат**:
```json
{
  "action_id": "CHECK_CASH_DETECTOR",
  "timestamp": "2026-03-28T13:20:00Z",
  "cashier_id": "Касса №4"
}
```

### Этап 4: Compliance Engine (Алёна)
**Вход**: Action ID из Этапа 3.
**Логика**: Сверка с `workflow_config.json`. Если действие нарушает порядок — генерация инцидента.
**Выход**: Объект Incident (JSON).
```json
{
  "id": "INC-20260328-001",
  "type": "Нарушение регламента",
  "level": "High",
  "description": "Пропущен этап проверки на детекторе",
  "screenshotUrl": "http://storage/frame_1240.jpg",
  "detectedObjects": ["Banknote", "Detector"]
}
```

### Этап 5: API и Интерфейс (Алексей)
**Вход**: POST запрос от Алёны на `/api/events/handle`.
**Действие**: 
1. Запись в Redis (для продакшена).
2. Рассылка через WebSocket всем открытым вкладкам.
3. Отображение карточки в ленте событий.

---

## 4. Дополнительные возможности (Metrics & Commands)

### Передача системных метрик (GPU/CPU)
Для отображения нагрузки во вкладке «Система», ML-сервер должен отправлять POST на `/api/metrics/system`:
```json
{
  "gpu_temp": 72,
  "gpu_load": 45.5,
  "vram_used": 4.2,
  "cpu_load": 12.0
}
```

### Обратная связь (Commands)
Интерфейс может отправлять команды на ML-сервер через бэкенд (POST `/api/ml/command`). 
Бэкенд перенаправит запрос на ваш `ml_endpoint/command`.
**Примеры**: `start_recording`, `reset_model_state`, `update_thresholds`.

---

## 5. Как проверить стыковку (End-to-End Test)

Для проверки всей цепочки "на столе" (offline):
1. **Макс** кладет видео в папку `/input`.
2. **Скрипт-интегратор** запускает последовательно все модули.
3. **Алёна** в конце своего модуля делает `requests.post` на URL вашего приложения.
4. **Алексей** (вы) подтверждает, что в браузере «всплыла» карточка именно по этому видео.

**Важно**: Все участники должны использовать одинаковые `action_id` (см. файл `workflow_config.json`).
